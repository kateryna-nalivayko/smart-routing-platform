# Як зараз реально працює оптимізація

У нас задача типу `VRPTW`:
- є техніки;
- є локації, які треба відвідати;
- у локацій є часові вікна;
- у візитів є тривалість;
- між точками є час на дорогу;
- не кожен технік підходить до кожної локації.

Тобто система намагається побудувати маршрути виду:

`офіс техніка -> локації -> офіс техніка`

## 2. Де знаходиться основна логіка

Основні файли:
- `app/adapters/optimization/excel_oasis_loader.py`
- `app/adapters/geocoding.py`
- `app/adapters/optimization/oasis_week_builder.py`
- `app/adapters/optimization/or_tools_solver.py`
- `app/adapters/optimization/solver_adapter.py`

## 3. Як проходить обробка даних

Все працює таким ланцюжком:

1. Береться Excel-файл `data/Routing_pilot_data_input_FINAL.xlsx`.
2. З нього читаються:
   - локації;
   - адреси;
   - техніки;
   - часові вікна;
   - тривалості робіт;
   - навички;
   - додаткові обмеження.
3. Адреси переводяться в координати через `data/geocoding_cache.json`.
4. На основі цих даних формується тижневий набір візитів.
5. Для кожного дня окремо запускається OR-Tools.
6. Після розв'язку формується:
   - JSON-лог;
   - Excel-таблиця;
   - короткий JSON з описом констрейнтів.

## 4. Як саме тут використовується OR-Tools

Перебирає допустимі варіанти маршрутів і шукає найкращий варіант в межах заданих обмежень.

По суті тут відбувається таке:

1. Кожен технік на конкретний день стає окремим `vehicle`.
2. Кожна локація/візит стає `node`.
3. Між усіма точками будується матриця часу в дорозі.
4. Для кожного візиту задається:
   - коли його можна почати;
   - скільки він триває;
   - які техніки взагалі мають право його виконувати.
5. Для кожного техніка задається:
   - початок зміни;
   - кінець зміни;
   - максимальний час роботи за день.
6. OR-Tools шукає таке призначення візитів, щоб:
   - не ламати часові вікна;
   - не ставити непідходящого техніка;
   - враховувати час на переїзд;
   - по можливості покрити максимум візитів.

## 5. Що було виправлено після зауважень

Раніше проблема була в тому, що в математичній частині були жорсткі заглушки:
- фіксований час між точками;
- фіксована відстань;
- фіксована сумарна дистанція;
- фіксована сумарна тривалість.

Зараз цього вже немає.

Що зроблено:
- прибрано фіктивні `15 хв` між будь-якими точками;
- прибрано фіктивні `5 км`, `50 км`, `300 хв`;
- додано реальний шар геокодування адрес через cache;
- час і відстань тепер рахуються від координат;
- додано нормальний вихідний JSON-лог;
- додано Excel-файл для людської перевірки результату.

## 6. Як зараз рахується дорога

Поточна версія рахує дорогу не через дорожній граф, а через координати.

Схема така:

1. Для адреси беруться `latitude/longitude`.
2. Між двома точками рахується Haversine distance.
3. Відстань переводиться в час через середню швидкість.

Тобто зараз це не Google Maps і не реальний навігатор.

Але це вже чесний базовий математичний розрахунок, а не заглушка.

## 7. Які констрейнти вже враховуються

### Враховуються

- Часові вікна локацій.
- Тривалість роботи на локації.
- Час на дорогу між точками.
- Навички техніка.
- Додаткові capability-ознаки:
  - фізично складна робота;
  - робота на висоті;
  - робота з ліфтом;
  - пестициди;
  - вимога громадянства.
- Permit-обмеження.
- Межі робочого дня техніка.
- Максимум робочого часу за день.
- Візити, де одночасно потрібно декілька техніків.

### Враховуються частково або у спрощеному вигляді

- `preferred technicians`
  Зараз це працює не як м'яке побажання, а фактично як allow-list.

- `avoid technicians`
  Зараз це жорстке виключення.

- Частота візитів протягом тижня
  Є розкладка по day-pattern, але не всі бізнес-правила формалізовані окремими обмеженнями.

## 8. Які констрейнти поки що не враховуються

- Перерва техніка як окремий обов'язковий інтервал у середині дня.
- Максимум годин на весь тиждень.
- Загальне правило "два візити не можна ставити день за днем".
- Старт і фініш з дому.
- Parking hub / walking cluster / мультимодальність.
- Повноцінний багатотижневий цикл на 2/3/4 тижні.

## 9. Який формат результату зараз є правильним

Основний файл:
- `data/output/<task_id>/schedule_log.json`

Кожен запис має поля:
- `date`
- `day_of_week`
- `start_time`
- `end_time`
- `technician_name`
- `location_name`
- `location_to`
- `activity_type`

Логіка така:

- якщо `activity_type = Commute`, то це переміщення;
- якщо `activity_type != Commute`, то це робота на локації;
- для сервісної роботи `location_to = null`.

Додатково формується:
- `data/output/<task_id>/schedule_log.xlsx`
- `data/output/<task_id>/constraints_summary.json`

## 10. Що треба для запуску без заглушок

Для нормального запуску потрібен файл:
- `data/geocoding_cache.json`

Якщо координат для якоїсь адреси немає, система:
- не підставляє фіктивні числа;
- не робить вигляд, що все порахувала;
- падає з явною помилкою про відсутні адреси.

Це спеціально зроблено так, щоб результат не був "красивим, але фальшивим".

## 11. Як запускати генерацію результату

Найпростіший варіант:

```bash
uv run python scripts/generate_schedule_artifacts.py --week-start 2026-03-16
```


## 12. Що показав останній запуск

Останній локальний запуск на тиждень, що починається `2026-03-16`, дав такий результат:

- побудовано `25` маршрутів;
- сформовано `405` рядків у `schedule_log.json`;
- `150` візитів залишилися непризначеними.

Це означає:
- solver реально працює;
- JSON та Excel реально генеруються;
- але поточний MVP ще не покриває всю предметну область на 100%.

## 13. Висновок по поточному стану

- OR-Tools у проєкті працює;
- фіктивні математичні заглушки прибрані;
- дані тепер можна читати через нормальний JSON-лог і Excel;
- частина важливих констрейнтів уже реалізована;
- частина констрейнтів поки що ще не реалізована і це треба прямо визнавати.
